# Permission-Discovery Harness — plan & handoff

> **Goal.** Build the infra as a principal that starts with **zero privileges**, let each
> Azure `AuthorizationFailed` reveal exactly which action/scope it needs, grant only that,
> and iterate until it applies clean — producing the **minimal RBAC set** the corporate
> deploy principal will need (where we won't have Owner). The separate RG + state is not the
> goal; it's the **blast-radius containment** that lets a no-privilege SP run safely without
> touching the real rig. This is a fresh-conversation workstream (long apply→grant loop).

## Existing Terraform state structure (verified 2026-06-30)
- **One monolithic remote state.** Single blob `poc0.tfstate` (~327 KB) in storage account
  `altoptfn95qtf`, container `tfstate`, RG `rg-altop-tfstate`, **AAD auth** (`use_azuread_auth=true`).
- **Partial backend.** `backend.tf` has `backend "azurerm" {}`; concrete values come from the
  **git-ignored `backend.hcl`** at init (`key = poc0.tfstate`). Generated by the `bootstrap/` root.
- **Two roots:** `infra/` (the main root, azurerm backend) and `infra/bootstrap/` (LOCAL state;
  one-time chicken-and-egg root that created the tfstate storage + emits `backend_hcl`).
- **Everything is in the one state:** root-level (`random_string.suffix`, subscription budget,
  `azurerm_resource_provider_registration.*`) + `module.network` + `module.dc[0]` +
  `module.app[0]` (substrate: ACR/CAE/Log Analytics/binder) + `module.quicksignals[0]` +
  `module.baldaydashboard[0]` (both via `modules/django_app`) + `module.frontdoor[0]`.
  Lifecycle-split by RG (`rg-altop-net/-onprem/-app`) and chunk-GATED (`deploy_*`), but NOT split
  by state file.
- **No `terraform_remote_state` anywhere.** Modules wire outputs in-process. The only `data`
  sources are three `azurerm_resource_group` lookups (create-or-reference RG pattern) — they read
  live Azure, not other states. → A split into platform + per-app states would be the corporate
  port's job (introduce `terraform_remote_state` then), not a same-state refactor.

## Harness design
1. **Zero-priv principal.** `az ad sp create-for-rbac` with **no role** (`--skip-assignment`). You
   (Owner / User Access Administrator) keep granting rights; the SP only ever creates resources.
2. **Auth wiring.** Point Terraform at the SP via `ARM_CLIENT_ID/SECRET/TENANT_ID/SUBSCRIPTION_ID`
   (replaces the current `use_cli`, which authenticates as *you*, an Owner — that's what masks the
   missing perms today). **First discovered grant:** the SP needs **`Storage Blob Data Contributor`**
   on the tfstate container just to `init`/read-write state (AAD-auth backend).
3. **Isolation.** New backend **key** (e.g. `disco.tfstate`) so it can't reconcile against
   `poc0.tfstate`; new **RG(s)** via a new `name_prefix` (e.g. `disco`); grant the SP its roles
   **scoped to those RGs only**.
4. **The loop.** `apply as SP → AuthorizationFailed (names the action + scope) → you grant the
   minimal role at that scope → re-apply → … → clean apply`. Parse the apply output (or Activity
   Log deny events) for `does not have authorization to perform action 'X' over scope 'Y'`. The
   accumulated grants ARE the answer.

## Gotchas that will define the corporate conversation
- **The infra self-creates role assignments.** The app modules make `azurerm_role_assignment`
  (AcrPull per app identity) → requires **`Microsoft.Authorization/roleAssignments/write`**, a
  high-privilege action corporate security often withholds. May force a design change (pre-create
  identities+assignments out-of-band, or a privileged sub-step). The harness surfaces it loudly.
- **Subscription-scoped + same-sub blind spot.** Provider registrations (`.../register/action`) and
  the consumption budget need **subscription** scope. But in the **same subscription** the RPs are
  **already registered**, so those actions no-op and **won't surface their permission requirement** —
  the harness under-reports them. Mitigate: **gate the RP-registration + budget resources off for
  discovery** (add a `manage_subscription_singletons` flag, default true, false for disco) and
  document their perms from the provider docs; or run discovery in a sub where they're unregistered.

## Deliverable
- A **custom role definition** JSON (`actions`/`dataActions`/`assignableScopes`) = the minimal role
  the corporate principal needs, plus
- A short list of **unavoidable high-privilege grants** (roleAssignments/write, sub-scoped
  registrations) flagged for the security team.
